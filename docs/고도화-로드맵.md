# OMR Grader 고도화 로드맵

작성일: 2026-07-24
대상 사용자: 교수, 조교
제품 제약: Windows 단독 실행 프로그램, 기존 입력·Excel 출력 흐름 유지, 서버·계정 기능 제외

## 1. 이번 고도화의 목표

현재 프로그램은 스캔 이미지와 정답키를 받아 채점 결과 Excel과 디버그 이미지를 만드는 데에는 충분하다. 다음 단계의 목표는 한 번의 채점 작업을 **검증 가능한 작업 기록**으로 바꾸는 것이다.

핵심 질문은 다음과 같다.

- 나중에 어떤 원본 이미지에서 나온 점수인지 다시 확인할 수 있는가?
- 정답키나 배점 기준이 바뀌었을 때 같은 응답으로 다시 분석·재채점할 수 있는가?
- 인식이 애매한 문항만 빠르게 찾아 사람이 확인할 수 있는가?
- 문항별 오답률·선택지 분포·난이도 신호를 보고 수업 개선에 활용할 수 있는가?
- 이 모든 과정이 기존 Windows 단독 실행 방식과 Excel 산출물을 깨지 않는가?

## 2. 현재 기준선

`omr_grader.py`는 단일 Python 파일 안에서 입력, 이미지 정렬, 마킹 인식, 채점, Excel 작성, 디버그 오버레이, Tkinter GUI까지 처리한다.

현재 확보된 기반:

- 이미지 파일과 다중 페이지 PDF를 입력으로 받음
- 90/180/270도 회전과 상하 반전 가능성을 보정함
- 학생 ID, 문항별 선택, 미응답·중복응답, 정렬 실패를 구분함
- 정답키는 xlsx/csv를 지원하고 행 수로 문항 수를 결정함
- `채점결과.xlsx`와 검토용 디버그 오버레이를 생성함
- GUI에서 폴더·파일 선택, 민감도 설정, 출력 폴더 선택, 백그라운드 채점을 지원함

현재 없는 핵심 기반:

- 채점 실행(run)과 원본 이미지의 영구 저장
- 문항별 인식 결과와 사용한 정답키·배점 기준의 버전 기록
- 이력 검색, 재열람, 재분석, 재채점
- 사람의 수정·검토 이력
- 학생별 결과 상세 화면과 문항별 통계 화면
- 대량 이미지 이력에서 썸네일·지연 로딩·페이지 검색

따라서 이 기능들은 서로 독립적인 메뉴로 추가하기보다, 먼저 **실행 기록과 원본 보존 모델**을 만들고 그 위에 연결하는 편이 안전하다.

## 3. 외부 사례에서 확인한 방향

| Gradescope | 문항 단위 재채점 요청, 상태 관리, 제출물 화면 안에서 검토·응답 | 학생별 전체 재채점보다 `문항/학생 단위 검토 큐`가 교수·조교 업무에 맞음 |
| Moodle | 성적·응답·통계·수동 채점 보고서, 변경 전 미리보기 가능한 재채점 | 재채점은 즉시 덮어쓰기보다 변경 예상값을 먼저 보여줘야 함 |
| Canvas New Quizzes | 전체 시험 통계, 문항 분석, 학생 분석, CSV 내보내기 | 총점만이 아니라 문항별 선택지 분포와 학생별 응답을 함께 제공해야 함 |
| Scantron | 평가 결과와 문항 분석·보고를 연결 | OMR 인식 결과를 보고서용 데이터로 재사용할 수 있는 구조가 필요함 |

## 4. 후보 기능 비교

점수는 1~5점이며 높을수록 유리하다. 구현 난이도와 위험은 높을수록 부담이 크다.

| 후보 | 교수·조교 가치 | 구현 난이도 | 기존 채점 호환성 | 데이터 안전 위험 | 확장성 | 핵심 산출물 | 검증 방법 |
|---|---:|---:|---:|---:|---:|---|---|
| A. 채점 이력·재열람 | 5 | 3 | 5 | 3 | 5 | 실행 목록, 학생 상세, 원본·오버레이 열기 | 동일 실행을 닫았다가 다시 열고 원본·Excel·상태가 일치하는지 확인 |
| B. 재분석·재채점 | 5 | 4 | 4 | 4 | 5 | 정답키/배점 버전, 변경 전후 미리보기, 새 결과 | 같은 응답에 새 키를 적용하고 변경 학생·문항·총점을 비교 |
| C. 검토 큐·수동 보정 | 5 | 3 | 4 | 4 | 5 | 미응답·중복·저신뢰 목록, 보정 사유·이력 | 애매한 마킹만 모아 원본 확인 후 재계산되는지 검증 |
| D. 문항·반 분석 리포트 | 4 | 3 | 5 | 2 | 5 | 선택지 분포, 오답률, 점수 분포, CSV/Excel 보고서 | 알려진 가상 결과와 집계값 대조 |
| E. GUI 작업 흐름 개선 | 4 | 3 | 5 | 1 | 4 | 채점 준비·이력·상세 화면 시안에 따른 UI | 대표 해상도, 키보드 이동, 긴 파일명, 오류 상태 수동 점검 |
| F. 성능·보안 기반 | 5 | 3 | 5 | 5 | 5 | 썸네일/지연 로딩, 검색·페이지 처리, 안전한 저장·삭제 | 대량 원본, 손상 파일, 권한 없는 경로, 백업 복원 시나리오 |
| G. 진단 로그·비정상 종료 복구 | 5 | 2 | 5 | 4 | 5 | 영구 로그, 실행 상태, 오류 상세, 중단 작업 감지·복구 안내 | 의도적 입력 오류, 강제 종료, 재실행 후 상태 복원 시나리오 |

### 해석

- A와 B는 사용자가 요청한 재열람·재분석·재채점 요구를 직접 충족하지만, C와 D의 기반이기도 하다.
- C는 현재 Excel을 사람이 직접 고쳐야 하는 상황을 줄이는 효과가 가장 크다. 다만 원본 보존과 변경 이력이 먼저 필요하다.
- D는 교수의 수업·문항 개선 가치가 높지만, 저장된 학생별 응답이 없으면 정확하게 만들 수 없다.
- E는 사용성 개선이지만 데이터 모델을 먼저 결정하지 않으면 화면만 바뀌고 기능이 분산될 수 있다.
- F는 별도 메뉴 하나라기보다 A~E에 반드시 포함해야 하는 공통 품질 기준이다.

## 5. 권장 의존 순서

사용자가 최종 기능을 선택하기 전의 기술적 의존 순서는 다음과 같다.

```text
실행 기록·원본 보존
        ├── 채점 이력·재열람
        │       ├── 재분석·재채점
        │       └── 검토 큐·수동 보정
        └── 문항·반 분석 리포트

GUI 시안 선택 ───────────────────────┘
성능·보안 기준은 모든 단계에 적용
```

이것은 기능 우선순위를 강제로 확정한 것이 아니라, 동일 데이터를 여러 번 저장하거나 화면별로 다른 결과를 만드는 위험을 줄이기 위한 구현 순서다. 최종 우선순위는 아래 시안과 후보 비교 후 선택한다.

## 6. 보존해야 할 데이터 단위

첫 구현에서 최소한 다음 연결을 보존하는 것을 기준으로 한다.

| 단위 | 보존 내용 |
|---|---|
| 채점 실행 | 실행 ID, 생성 시각, 입력 목록, 상태, 오류·경고 요약 |
| 원본 | 원본 파일 경로 또는 앱 저장본, 파일명, 크기, 해시, 페이지 정보 |
| 인식 결과 | 학생 ID, 문항별 선택, 미응답·중복 상태, 검토 필요 여부 |
| 정답키 버전 | 원본 파일명, 내용 해시, 문항 수, 정답, 생성 시각 |
| 채점 기준 | 문항별 배점 또는 기본 배점, 반올림 규칙, 기준 버전 |
| 결과 버전 | 적용한 정답키·기준, 점수, 변경 전후 차이, 내보내기 시각 |
| 검토 이력 | 검토자 입력값, 사유, 이전 값, 새 값, 검토 시각 |
| 진단 기록 | 실행 ID, 처리 단계, 시각, 수준, 오류 유형, 파일 식별자, 상세 예외 정보 |
| 복구 상태 | 시작·진행·완료·실패·중단 상태, 마지막 처리 항목, 재시작 가능 여부 |

원본 이미지는 자동으로 버리지 않는다. 단, 저장 위치·백업·삭제는 사용자가 확인할 수 있어야 하며, 원본을 앱 데이터 폴더에 복사할지 기존 경로를 참조할지는 구현 시 저장 용량과 이동식 디스크 사용성을 비교한다.

## 7. 성능·보안 설계 기준

### 성능

- 원본은 원본대로 보존하되 목록에서는 축소 썸네일만 사용한다.
- 상세 화면을 열 때만 원본 또는 고해상도 오버레이를 지연 로딩한다.
- 이력 목록은 검색·필터·페이지 단위로 읽어 수천 장의 이미지를 한 번에 화면에 만들지 않는다.
- 기존 백그라운드 채점 작업을 유지하고, 진행률·취소 가능성·실패 항목을 실행 단위로 표시한다.
- Excel과 썸네일 생성은 인식과 분리해, 인식 완료 결과를 잃지 않도록 한다.

### 진단·복구

- 현재 GUI에 표시되는 활동 메시지와 별도로 파일 기반 진단 로그를 남긴다.
- 오류 로그에는 실행 ID, 처리 단계, 입력 파일 식별자, 오류 유형, 상세 예외 정보를 포함한다. 원본 답안 이미지 자체는 로그에 복사하지 않는다.
- 일반 사용자에게는 이해 가능한 오류 요약을 보여주고, `진단 로그 폴더 열기`로 상세 로그를 확인할 수 있게 한다.
- 채점 실행은 시작·진행·완료·실패·중단 상태를 저장한다. 프로그램이 비정상 종료되면 다음 실행 시 `중단된 작업을 발견했습니다`를 표시한다.
- 중단 작업은 자동으로 성공 처리하지 않는다. 가능한 경우 마지막 처리 항목부터 재시작하고, 불가능하면 원본과 부분 결과를 보존한 채 새 실행으로 재시도하게 한다.
- 로그 파일은 민감한 학생 응답이나 원본 이미지 내용을 남기지 않고, 경로·해시·학생 식별자는 필요 최소한으로 기록한다.
- 로그 크기는 상한과 보관 기간을 두어 무한히 쌓이지 않게 하며, 사용자가 로그를 압축해 지원·디버깅용으로 내보낼 수 있게 한다.

### 보안·안전

- 이미지·CSV·XLSX·PDF는 확장자만 믿지 않고 읽기 단계에서 형식과 크기를 검증한다.
- 저장·백업 경로는 사용자가 선택한 범위 안에서만 다루고, 프로그램이 관리자 권한을 요구하지 않도록 한다.
- 삭제는 실행 기록과 원본을 구분해 확인하며, 기본 동작은 휴지통 또는 복구 가능한 백업을 고려한다.
- Excel 내보내기 셀에는 입력 파일명이 수식으로 해석되지 않도록 문자열 안전 처리를 적용한다.
- 로컬 파일 접근 오류·손상 이미지·권한 오류를 사용자 메시지로 남기고, 부분 결과를 성공으로 표시하지 않는다.
- 백업 파일에는 실행 기록과 원본 연결 정보가 포함되어야 하며, 복원 테스트를 제공한다.
- Windows 사용자 데이터 영역에 저장하고, 시스템 전체나 임의의 상위 폴더에 쓰지 않는 최소 권한 원칙을 적용한다.

현재 의존성 제한이 있으므로 암호화 라이브러리를 새로 추가하지 않는 것을 기본으로 한다. 저장 암호화가 필수 요구가 되면 Windows 보호 저장소 사용 가능성과 의존성 추가 여부를 별도 결정한다.

## 8. 단계별 제안

### 단계 0: 선택 전 검토

- 이 문서의 후보 비교 확인
- 아래 3개 GUI 방향 중 선호 방향 선택
- 첫 구현 범위와 성공 기준 확정
- 오류 로그·비정상 종료 감지 기준 확정

### 단계 1: 공통 기록 기반

- 채점 실행 ID와 상태
- 원본·정답키·기준의 연결
- 학생별 문항 응답과 결과 저장
- 실행 상태와 영구 진단 로그 저장
- 비정상 종료 감지 및 재실행 안내
- 기존 `채점결과.xlsx` 출력은 계속 생성

### 단계 2: 선택 기능 구현

선택 가능한 첫 기능은 다음과 같다.

1. **이력·재열람 우선**: 가장 안전하게 운영 기반을 확보
2. **재분석·재채점 우선**: 정답키 변경과 점수 기준 변경 문제를 먼저 해결
3. **검토 큐 우선**: 미응답·중복·애매한 마킹의 사람 확인을 먼저 개선
4. **문항 분석 우선**: 교수의 수업 개선 보고서를 먼저 제공

### 단계 3: 연결 기능

- 검토 결과를 재채점에 반영
- 문항·반 통계를 동일 이력에서 생성
- 결과 상세에서 원본, 오버레이, 이전 결과, 새 결과를 비교
- 선택한 GUI 방향에 맞춰 검색·필터·내보내기 흐름 정리

## 9. 현재 확정된 GUI 방향과 전체 구현 순서

다음 단계에서 아래 두 가지를 함께 선택하면 바로 설계·구현으로 넘어갈 수 있다.

- 기본 구조: C `교수용 요약 워크스페이스`
- 결과 상세: B `증거 중심 검토 콘솔`의 원본·인식·정답키 비교와 수동 채점 적용
- 공통 안전장치: 실행 상태, 영구 진단 로그, 비정상 종료 감지·복구 안내
아래 기능은 모두 구현 대상이다. 단계는 기능을 제외하기 위한 우선순위가 아니라, 데이터 의존성과 오류 복구 위험을 줄이기 위한 구현 순서다.

1. 이력·원본 보존·진단 로그·중단 작업 감지
2. 재열람·재분석·재채점과 변경 전후 미리보기
3. 검토 큐와 수동 채점 적용·검토 이력
4. 문항·반 분석과 Excel/CSV 보고서
5. C+B 통합 GUI의 검색·필터·상세·내보내기 연결

각 단계가 끝나도 기존 단일 채점과 `채점결과.xlsx` 출력은 계속 동작해야 하며, 모든 단계에 성능·보안·진단 로그 기준을 적용한다.
